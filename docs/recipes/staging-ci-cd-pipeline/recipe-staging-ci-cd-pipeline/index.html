Staging CI/CD on GitLab and Google App Engine

In this recipe, you will be guided through the process of creating a CI/CD pipeline on GitLab that tests, builds and deploys next-with-moxy based projects on Google App Engine into a staging environment.

#Our Requirements

A few strict requirements were set for our CI/CD pipeline:

  1. Be easy to setup and maintain.
  2. Be cost effective.
  3. Have an environment as similar as possible to the production environment.
  4. Have a staging environment accessible publicly via a URL, like https://some-domain.com, continuously deployed of master branch.
  5. Have a environment for each release accessible publicly via a URL https://v1-4-5.some-domain.com, continuously deployed of release tags.
  6. Have a preview environment for every merge request accessible via publicly a URL like https://feat-homepage-hero.some-domain.com, continuously deployed of the branch associated with the merge request.
  7. Have HTTPS enabled for all URLs.

We have successfully used Zeit's Now in the past and it ticked all the above requirements. But with the v2 release, Now enforces serverless lambda deployments which are different from regular deployments that use the npm start command. This effectively broke point 3 of our requirements list.

After analyzing a lot of alternatives from various providers against our requirements, we settled on using Google App Engine from Google Cloud Platform.

#Walk-through

The steps below can only be completed by Tech Leads due to permissions.

#1. Create the project in Google Cloud Platform

  1. Login into Google Cloud Console using your MOXY's email.
  2. Create a new project by clicking in the header's dropdown and select New project.
  3. Fill in the Project Name and Project ID fields by using the following convention: 3.1. Project Name must be equal to the GitLab group name, like acme. 3.1. Project ID should be equal to the Project Name but suffixed with -with-moxy, like acme-with-moxy. Since the Project ID will determine the your application URLs, you may use an abbreviation of the Project Name when composing the Project ID if it's too large.
  4. Once created, select the recently created project in the header dropdown.

#2. Create the application in Google App Engine

  1. Click in App Engine menu item on the left sidebar and then hit Create Application.
  2. Choose europe-west as the region.
  3. Choose NodeJS as the language and Standard as the environment.
  4. Wait until the application is created.
  5. Once done, you may skip the wizard's next steps (I'll do this later).

#3. Enable Cloud Build API

For GitLab to be able to interact with App Engine, we must enable App Engine Admin API:

  1. Click in API & Services > Library menu item on the left sidebar.
  2. Search for Cloud Build API and then click Enable.

#4. Enable App Engine Admin API

For GitLab to be able to interact with App Engine, we must enable App Engine Admin API:

  1. Click in API & Services > Library menu item on the left sidebar.
  2. Search for App Engine Admin API and then click Enable.

#5. Create an account for GitLab

Now that we enabled App Engine Admin API, we must create an account for GitLab to be able to interact with it:

  1. Click in IAM & Admin > Service accounts menu item on the left sidebar and then hit Create Service Account on the top.
  2. Type gitlab-ci in the Service Account Name and hit Create.
  3. Add App Engine Deployer, App Engine Service Admin, Cloud Build Editor, Storage Admin roles and hit Continue.
  4. Click in the Create Key button, choose JSON and hit Create. The key should have been downloaded to your computer.

#6. Add .gitlab-ci.yml

Copy .gitlab-ci.yml into the root folder of your project. You may take a look at it to understand the different stages and jobs.

#7. Add .gcloudignore

Copy .gcloudignore into the root folder of your project. This file contains a blacklist of patterns that will match files that will not be uplodated to App Engine.

#8. Setup CI/CD environment variables

The .gitlab-ci.yml file references some environment variables that you must provide.

  1. On the GitLab repository page, click Settings -> CI/CD.

  2. Expand the Variables section and fill the following variables:

    Contents of app.yml:

    runtime: nodejs12
    env: standard
    instance_class: B1
    basic_scaling:
      max_instances: 1
      idle_timeout: 5m
    handlers:
    - url: /.*
      secure: always
      redirect_http_response_code: 301
      script: auto

    The app settings above makes sure that we are cost effective by using the cheapest instance class and by having a maximum of 1 instances running at all times. Moreover, that instance will automatically dispose after 5 minutes of idle time, and will spawn again on a new request, taking a maximum of 10 secs to wake up. Please note that we are only charged during the time the instance is up and not while it's disposed! üëå

#9. Test it out!

You may now commit .gitlab-ci.yml into a new branch, push it and open a merge request to test things out. Be sure to also check if the preview environment was successfully un-deployed when the merge request got merged.
